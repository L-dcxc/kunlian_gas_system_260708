# 气体安全报警监控系统功能实现方案

> 本文档用于指导 AI Agent 全权实现本软件。实现前必须同时阅读目录中的需求文档和协议文档，本文档作为架构设计、模块拆分、实现约束和开发顺序的统一说明。

## 1. 项目目标

开发一套 Windows 桌面端气体安全报警监控系统上位机软件，用于通过 RS485 串口或 485 转 LAN 网络方式接入气体报警控制器或探头设备，完成实时数据采集、报警监控、地图点位展示、曲线分析、历史记录、用户权限、数据备份恢复、本地 API、大屏展示和联动控制等功能。

软件需要支持两套 Modbus RTU 协议，但同一套部署或同一项目配置中只使用其中一套协议，不需要同时混合接入两套协议设备。

## 2. 推荐技术栈

若从零开发，推荐使用以下技术栈：

- 桌面框架：`PySide6` 或 `PyQt6`
- 数据库：`SQLite`
- 串口通信：`pyserial`
- Modbus/CRC：可使用 `pymodbus`，也可自实现 RTU 组帧、拆帧和 CRC16
- 本地 API：`FastAPI` 或 `Flask`
- 定时任务：`APScheduler`
- 曲线图表：`pyqtgraph` 或 Qt Charts
- Excel 导入导出：`openpyxl`
- PDF/打印：Qt 打印能力或 HTML 模板转 PDF
- 打包：`PyInstaller` 或 `Nuitka`

设计原则：

- 设备采集、协议解析、业务状态、界面展示必须分层。
- 协议差异必须隔离在协议适配层，界面和业务逻辑只使用统一设备数据模型。
- 所有关键操作必须记录操作日志。
- 所有实时状态必须能落库或从内存状态中心读取。
- 软件优先保证稳定、可维护、可现场排障，不追求复杂动画效果。

## 4. 总体架构

推荐架构如下：

```text
桌面 UI 层
  登录/监控/地图/设备/曲线/记录/配置/大屏

应用服务层
  用户服务
  设备配置服务
  采集任务服务
  报警状态服务
  地图点位服务
  记录查询服务
  备份恢复服务
  授权服务
  本地 API 服务

设备采集层
  串口通道
  TCP 通道
  轮询调度
  超时与重连
  原始报文日志

协议适配层
  协议 1 适配器
  协议 2 适配器
  统一设备状态模型

数据持久层
  SQLite
  地图文件
  备份文件
  配置文件
```

核心要求：

- UI 不直接访问串口或 TCP。
- UI 不直接解析 Modbus 报文。
- 采集层只负责收发字节、轮询、超时。
- 协议层负责把原始寄存器转换为统一模型。
- 业务层负责报警、记录、联动、地图状态同步。

## 5. 统一设备状态模型

无论协议 1 还是协议 2，最终都必须转换为统一数据结构：

```python
DeviceReading:
    protocol: str                # protocol_1 / protocol_2
    source_type: str             # controller / probe
    port_id: int
    controller_id: int | None
    detector_id: int
    controller_address: int | None
    detector_address: int | None
    register_index: int | None
    status: str                  # normal / alarm_low / alarm_high / fault / offline / disabled / over_range / warming
    concentration: float | None
    gas_type: str | None
    unit: str | None
    alarm_level: int | None
    raw_status: int | str | None
    raw_value: str | None
    timestamp: datetime
```

业务层和界面层只能依赖该统一模型，不允许直接依赖某一套协议的寄存器定义。

## 6. 协议模式设计

系统配置中必须提供协议模式选择：

- 协议 1
- 协议 2

同一运行项目只允许选择一种协议模式。切换协议模式后，应提示用户重新启动采集服务或重启软件。

协议适配器接口建议：

```python
class ProtocolAdapter:
    def build_poll_requests(config) -> list[PollRequest]:
        pass

    def parse_response(request: PollRequest, response_bytes: bytes) -> list[DeviceReading]:
        pass

    def validate_response(request: PollRequest, response_bytes: bytes) -> bool:
        pass
```

协议 1 与协议 2 分别实现该接口。

注意：

- 协议 1 和协议 2 的寄存器数量、状态定义、浓度格式可能不同。
- CRC 字节序以真实协议文档和实测设备为准。
- 需要在设备调试页面显示原始发送报文、原始返回报文、解析结果和错误原因。

## 7. 通讯通道设计

系统需要支持两类通道：

### 7.1 RS485 串口

配置项：

- 端口名称，例如 COM1、COM3
- 波特率
- 数据位
- 停止位
- 校验位
- 采集周期
- 超时时间
- 连续失败次数阈值
- 重连间隔

### 7.2 TCP 网络

配置项：

- IP 地址
- 端口
- 连接超时
- 读超时
- 采集周期
- 连续失败次数阈值
- 重连间隔

说明：

- TCP 场景通常是 485 转 LAN，默认按 RTU over TCP 处理，即 TCP 中传输 Modbus RTU 帧。
- 若后续甲方设备是 Modbus TCP，需要单独增加 Modbus TCP 适配。

## 8. 采集任务设计

采集任务必须支持启动、停止、重启和异常恢复。

基本流程：

```text
读取端口配置
读取控制器/探测器配置
根据协议模式生成轮询请求
打开串口或 TCP
循环发送请求
等待响应
校验响应
协议解析
更新实时状态中心
触发报警状态机
按存储周期写入运行记录
异常时累计失败次数
超过阈值后标记离线
按策略重连
```

要求：

- 采集线程不得阻塞 UI。
- 多端口采集应互不影响。
- 单个设备超时不应导致整个系统卡死。
- 需要提供采集服务状态：未启动、运行中、异常、重连中、已停止。

## 9. 报警状态机

报警状态不能只靠界面颜色判断，需要独立状态机。

每个探测器维护当前状态：

- 正常
- 低报
- 高报
- 超量程
- 故障
- 离线
- 屏蔽
- 预热

状态变化时执行：

```text
正常 -> 报警：新增报警记录，弹窗，警情列表新增，地图/卡片闪烁，触发联动
报警 -> 报警持续：更新持续状态，不重复新增同一条未恢复报警
报警 -> 正常：记录恢复时间，停止闪烁，必要时保留警情
正常 -> 故障：新增故障记录和提示
任意 -> 离线：标记离线，界面置灰，记录通讯异常
离线 -> 正常：恢复在线状态
```

必须避免：

- 同一个报警周期重复写入大量报警记录。
- 设备恢复后界面仍显示报警。
- 弹窗无限重复。
- 离线和报警状态互相覆盖不清。

建议表设计中报警记录包含：

- detector_id
- alarm_type
- alarm_level
- trigger_value
- start_time
- end_time
- status: active / recovered
- confirm_status 可预留

## 10. 地图点位设计

地图模块包括：

- 地图上传
- 地图删除
- 地图切换
- 探测器点位绑定
- 点位拖拽
- 点位坐标保存
- 状态实时显示
- 报警闪烁
- 点击查看详情

坐标保存必须使用比例坐标：

```text
x_ratio = 点位中心 X / 地图显示宽度
y_ratio = 点位中心 Y / 地图显示高度
```

不要只保存固定像素坐标。这样才能适配窗口缩放、全屏、大屏和不同分辨率。

地图文件建议保存到应用数据目录，例如：

```text
data/maps/
```

数据库只保存文件路径、原始文件名、绑定关系和比例坐标。

## 11. 曲线监控设计

曲线模块包括：

- 实时曲线
- 历史曲线
- 明细列表
- 时间范围筛选
- 通讯端口筛选
- 控制器筛选
- 探测器筛选
- 缩放和平移
- 打印或导出

实时曲线读取内存状态中心和短期缓存。

历史曲线读取运行记录表。

建议：

- 实时曲线只保留最近一段时间数据，例如最近 10 分钟或最近 N 个点。
- 历史曲线按时间范围查询，避免一次加载过大数据。

## 14. 用户与权限设计

角色：

- 管理员：唯一，拥有全部权限。
- 操作员：可多个，只能查看监控页面和查询数据，无系统设置权限，无退出系统权限。

必须实现：

- 登录
- 修改密码
- 用户新增、修改、删除
- 管理员唯一约束
- 菜单权限控制
- 关键操作权限控制
- 操作日志

注意：

- 不允许创建第二个管理员。
- 删除管理员前必须有保护逻辑，避免系统无管理员。
- 初始管理员账号需要初始化方案。

## 15. 软件授权设计

一机一码授权建议流程：

```text
首次启动 -> 生成机器码 -> 用户提交机器码 -> 输入授权码 -> 校验通过 -> 保存授权状态 -> 正常使用
```

机器码可基于：

- 主板序列号
- CPU ID
- 磁盘序列号
- 网卡 MAC

授权信息保存时应避免明文直接存储，可使用签名或加密校验。

未授权状态：

- 可显示授权窗口。
- 不允许进入主系统。
- 或只允许进入有限演示模式，具体以甲方规则为准。

## 16. 本地 API 设计

本地 API 为只读接口。

建议接口：

```text
GET /api/health
GET /api/devices/realtime
GET /api/devices/{id}/realtime
GET /api/alarms/active
GET /api/alarms/history
GET /api/controllers
GET /api/detectors
```

返回格式统一：

```json
{
  "success": true,
  "code": 0,
  "message": "ok",
  "data": {}
}
```

要求：

- API 服务随软件启动，可在设置中启停。
- API 端口可配置。
- 端口占用时给出明确提示。
- 接口字段需要形成说明文档。

## 17. 备份与恢复设计

### 17.1 手动备份

备份内容建议包括：

- SQLite 数据库
- 地图文件
- 配置文件
- 授权文件是否纳入备份需单独配置

备份文件建议使用压缩包：

```text
backup_YYYYMMDD_HHMMSS.zip
```

### 17.2 定时备份

配置项：

- 是否启用
- 备份周期
- 备份时间
- 备份目录
- 保留份数
- 失败提醒

### 17.3 数据恢复

恢复流程：

```text
选择备份文件
校验备份结构
提示当前数据将被覆盖
停止采集服务
执行恢复
恢复成功后重启软件或重载数据
失败时提示原因
```

恢复前建议自动生成一份当前数据的安全备份。

## 18. 联动控制设计

联动控制包括：

- 联动对象管理
- 联动规则配置
- 自动联动触发
- 手动控制
- 联动记录

联动对象类型：

- 风机
- 电磁阀
- 继电器
- 声光报警器
- 分区设备

规则条件：

- 探测器
- 报警类型
- 报警级别
- 联动对象
- 触发延时
- 恢复动作
- 是否启用

自动联动必须与报警状态机配合，避免同一报警持续期间重复触发。

外部 IO 继电器控制需要甲方提供协议和点表。若协议暂未提供，先实现联动对象、规则、模拟触发和联动记录，真实下发指令作为独立适配器预留。

## 19. 大屏展示设计

大屏模式包括：

- 全屏展示
- 数据监控大屏
- 地图监控大屏
- 设备状态大屏
- 页面轮播
- 报警优先展示

要求：

- 全屏后信息完整。
- 页面轮播时间可配置。
- 发生报警时可自动切换到报警页面或对应地图点位。
- 大屏页面应使用更高对比度和更大的字体。

## 20. 设备调试页面

设备调试页面用于现场排障，必须尽量实用。

功能包括：

- 选择端口
- 选择协议
- 选择设备地址
- 手动发送读请求
- 显示发送报文 HEX
- 显示返回报文 HEX
- 显示 CRC 校验结果
- 显示解析后的状态、浓度、单位
- 显示错误原因，例如超时、CRC 错误、长度错误、未知状态

该页面对现场联调非常重要，应优先实现基础版本。

- 协议适配说明。
- 简要用户操作说明。

## 26. 开发注意事项

- 不要把协议解析写在 UI 代码中。
- 不要把串口读写阻塞在主线程。
- 不要用固定像素保存地图点位。
- 不要让报警记录在同一报警周期内无限重复写入。
- 不要让数据恢复在采集运行中直接覆盖数据库。
- 不要让操作员看到或执行系统设置类操作。
- 不要将授权信息以纯明文方式保存。
- 不要在没有确认协议的情况下写死波特率、寄存器数量和状态含义。

## 27. 默认假设

若甲方未进一步说明，按以下默认规则实现：

- 软件运行平台：Windows 10/11。
- 数据库：本地 SQLite。
- API：本机只读 HTTP API。
- 地图点位：比例坐标保存。
- 协议模式：系统设置中选择协议 1 或协议 2。
- TCP：默认按 RTU over TCP。
- 报警记录：同一探测器同一报警类型，未恢复前不重复生成新报警记录。
- 备份：包含数据库、地图文件和配置文件。
- 管理员：系统只允许存在一个管理员。
- 操作员：不可进入系统设置，不可退出系统。
